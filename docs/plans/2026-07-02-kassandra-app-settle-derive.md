## Feasibility (confirmed — the "not derivable" note was overcautious)
The `settleChallenge` builder (`sdk/src/instructions/challenge.ts:182-240`) derives oracle/market/stakeVault/escrowVault internally from `{oracleNonce, aiClaim}`; the caller supplies 15 accounts, and EVERY one is recoverable:
- **Directly on the decoded Market** (`sdk/src/accounts/market.ts`): `aiClaim`, `proposer`, `question`, `passAmm`, `failAmm`, `kassVault`, `usdcVault`, `oraclePassKass`, `oracleFailKass`, `challenger`, `challengerUsdcVault`.
- **Derived (CU3 already does all of these — reuse `app/src/data/actions/challengeCompose.ts`'s helpers):**
  - `passKassMint`/`failKassMint` = `conditionalTokenMint(market.kassVault, 0/1)` (CU2's `conditionalTokenMint`, pass=0/fail=1).
  - `kassVaultUnderlying` = the KASS conditional vault's underlying token account (CU3 knows this from `composeVault`/`initializeConditionalVault` — the vault-underlying PDA/ATA; confirm the exact derivation from the SDK/compose).
  - `cvEventAuthority` = `futarchy.pda.vaultEventAuthority()` (fixed PDA; CU3 uses it).
  - `proposerUsdc` = `ATA(market.proposer, oracle.usdcMint)`.
  - `challengerUsdcDest` = `ATA(market.challenger, oracle.usdcMint)` (OR `market.challengerUsdcVault` — confirm which the program wants for the DEST vs the escrow; RF4's `buildSettleChallengeIxs` arg names `challengerUsdcDest` vs the escrow `challengerUsdcVault` — get this right from the settle handler/the challenge-market-e2e settle call).
  - `challengerKass` = `ATA(market.challenger, oracle.kassMint)`.
- The `oracleNonce` is the same nonce the finalize/challenge UI already recalls (`app/src/lib/nonceStore.ts` `recallNonce` + the `resolveOracleNonce` scan) — reuse it.

So a `buildSettleFromMarketIxs({ connection, oracleNonce, market, oracle })` can derive all 15 + call `buildSettleChallengeIxs` — no JSON paste.
