# Kassandra Functional dApp — Wallet + Read-Only Browse (slice 2) — Design + Plan

> **For Claude:** REQUIRED SUB-SKILL: subagent-driven-development (per-task implement + review).

**Goal:** Turn the slice-1 UI foundation into a functional dApp's first read slice: wire **wallet-adapter** (real Connect) + a configurable **RPC provider**, then the **read paths** — an oracle **browser** (enumerate + decode all oracles) and an oracle **detail** view (its facts, proposers, AI claims, market — all decoded via `@kassandra/sdk`). No signing yet (write flows = next slice). Prove the data layer against a **seeded surfpool/local validator** (deploy + seed real oracles → drive the read paths). All in the Delphi visual language from slice 1.

## Decisions (locked with the user)
1. **Configurable RPC; prove against a seeded surfpool/local validator.** The app reads `VITE_RPC_URL` (+ a UI cluster selector: localnet/surfpool/devnet/mainnet), default localnet. The program has NO standing deployment — a gated integration test deploys+seeds oracles on surfpool/a local validator and drives the read data layer against real on-chain bytes (the harness pattern used throughout). The UI is correct wherever it points.
2. **First functional slice = wallet + read-only browse.** Real wallet connect + RPC provider + oracle list + oracle detail (all decoded). No signing. Write flows (create/propose/vote/claim) = the NEXT slice.
3. Builds on slice 1 (`app/`, Vite+React+Tailwind-v4, the Delphi primitives + tokens). **Build the SDK first** (`pnpm --filter @kassandra/sdk build`) — the app resolves `@kassandra/sdk` types from `sdk/dist` (gitignored) — this slice DEPENDS on the SDK for real (decoders + PDA derivers), unlike slice 1's link-proof.

## Source of truth
- **SDK read surface:** `sdk/src/accounts/{oracle,fact,proposer,aiClaim,market,factVote,protocol}.ts` — `decodeX(data: Uint8Array): X` + the typed interfaces (Oracle exposes creator/kassMint/usdcMint/stakeVault/deadline/phase/phaseEndsAt/optionsCount/proposerCount/survivingCount/factCount/bondPool/settledCount/resolvedOption/openChallengeCount/thresholds/etc.). `sdk/src/accounts/common.ts` — `ACCOUNT_TYPE_OFFSET` (the account_type tag byte) + `assertAnchorAccount`/the tag check; `sdk/src/constants.ts` — `AccountType` enum, `ACCOUNT_SIZES`, `Phase` enum, `KASSANDRA_PROGRAM_ID`. `sdk/src/pda.ts` — the PDA derivers (oracle/fact/proposer/aiClaim/market/protocol). NOTE: the TS SDK has NO getProgramAccounts helper (the runner had one in Rust) — the app implements enumeration: `connection.getProgramAccounts(KASSANDRA_PROGRAM_ID, { filters: [{ memcmp: { offset: ACCOUNT_TYPE_OFFSET, bytes: <base58 of the tag byte> } }] OR { dataSize: ACCOUNT_SIZES.Oracle }] })` → `decodeOracle(acct.data)`.
- **web3.js:** `@solana/web3.js` 3.0.0-rc.2 (classic-style `Connection`, `PublicKey`, `getProgramAccounts`, `getAccountInfo`, `getMultipleAccountsInfo`). Wallet-adapter deps already installed in slice 1 (`@solana/wallet-adapter-react`/`-base`/`-react-ui`/`-wallets`) — NOT yet wired.
- **Slice-1 foundation to reuse:** `app/src/components/ui/*` (Card, EyebrowTag, SectionHeader, Button, AvatarBubble, TriggerPreviewCard, etc.), the `@theme` tokens (`app/src/index.css`), the router (`app/src/main.tsx` — add `/oracles` + `/oracles/:pubkey`), `app/src/components/landing/NavBar.tsx` (the Connect placeholder → make it real). The surfpool harness pattern: `sdk/test/surfpool/harness.ts` (deploy the program via setAccount, fabricate SPL, seed accounts, RPC).
- **Seeding real oracles:** reuse how the SDK E2Es create oracles (`sdk/test/surfpool/lifecycle-e2e.test.ts` / `settlement-e2e.test.ts` — `initProtocol` + `createOracle` + `propose` etc.) to seed a few oracles in varied phases for the browse test.

## Tasks

### FA1 — Wallet-adapter + RPC provider + app shell
- **RPC/cluster provider** (`app/src/providers/` or `app/src/lib/`): a `ConnectionProvider`-backed context reading `VITE_RPC_URL` (default `http://127.0.0.1:8899`) with a UI **cluster selector** (localnet / surfpool[same localnet default] / devnet / mainnet-beta — endpoints configurable; persist the choice in localStorage). Expose a typed `useConnection()`/`useCluster()`.
- **Wallet-adapter:** wrap the app in `ConnectionProvider` + `WalletProvider` (+ `WalletModalProvider` from `-react-ui`) with a sensible wallet set (e.g. Phantom/Solflare via `-wallets`, or the standard-wallet autodetect). Make the NavBar **Connect** button REAL (open the wallet modal / show the connected address + a disconnect) — restyle the wallet-adapter button to the Delphi NavPill look (soft-cream, sepia, 12px) OR wrap the adapter's connect in the existing NavPill. Import the adapter's CSS but override to match Delphi (no default purple).
- **App shell:** the router gains authenticated-agnostic routes; a shared layout (the NavBar + a footer) wraps `/`, `/oracles`, `/oracles/:pubkey`, `/styleguide`. The landing stays at `/`. Keep it read-only (no tx yet). No console errors; the connect flow works (verify with the wallet modal opening — a real wallet needn't be installed to prove the modal + provider wiring).
- `pnpm --filter @kassandra/sdk build` (SDK first) + `pnpm --filter app typecheck` + `build` (green + `verify-css OK`) + `lint`. Commit `feat(app): wallet-adapter + configurable RPC provider + app shell`.

### FA2 — Oracle read data layer + gated surfpool integration test
- **Data layer** (`app/src/data/` — pure, testable functions taking a `Connection`, NO React): 
  - `fetchOracles(connection): Promise<{ pubkey, oracle: Oracle }[]>` — `getProgramAccounts(KASSANDRA_PROGRAM_ID, { filters: [account_type == Oracle] })` (memcmp on `ACCOUNT_TYPE_OFFSET`, and/or `dataSize: ACCOUNT_SIZES.Oracle`), decode each with `decodeOracle`, skip/annotate any that fail the tag/size check. Sort (e.g. by deadline desc). 
  - `fetchOracleDetail(connection, oraclePubkey): Promise<{ oracle, facts: Fact[], proposers: Proposer[], aiClaims: AiClaim[], market?: Market }>` — fetch the Oracle, then enumerate its children (getProgramAccounts filtered by account_type + a memcmp on the child's `oracle` field offset — determine each child's oracle-field offset from the SDK interface/layout, mirroring the runner's Fact.oracle-offset approach; OR derive known PDAs where deterministic). Decode all.
  - Handle RPC errors + empty results gracefully (typed results, no throws that crash the UI).
- **Gated integration test** (`app/test/oracle-data.e2e.test.ts` or under a gated dir; `KASSANDRA_E2E=1`): boot surfpool/a local validator, deploy the program, `initProtocol`, seed 2–3 oracles in varied phases (+ a fact/proposer/aiClaim on one) using the SDK builders (mirror the SDK E2E seeding). Run `fetchOracles` + `fetchOracleDetail` against the real cluster over RPC → ASSERT the decoded oracles match what was seeded (count, phases, the child facts/proposers/claims). This proves the read data layer against REAL on-chain bytes. Keep the default `pnpm --filter app test` (if any) offline; gate the surfpool test. (Reuse `sdk/test/surfpool/harness.ts` — import it, or replicate the minimal deploy+seed.)
- Unit tests (offline): `fetchOracles`/`fetchOracleDetail` against a MOCK connection (canned getProgramAccounts/getAccountInfo returning real decoded-account byte fixtures) — assert enumeration + decode + the tag-mismatch skip. 
- `pnpm --filter app typecheck` + `build` + `lint` green + the offline unit tests pass. Commit `feat(app): oracle read data layer + gated surfpool integration test`.

### FA3 — Browse UI: oracle list + oracle detail (Delphi style)
- **Oracle list** (`app/src/pages/Oracles.tsx` at `/oracles`): consume `fetchOracles` via a small query hook (loading/error/empty states). Render each oracle as a Delphi `Card` in a responsive grid: the question/prompt (or a prompt-hash short + a human label if we only have the hash — show the hash truncated + the phase), an `EyebrowTag`/status chip for the `Phase` (Proposal/Fact/AiClaim/Challenge/Resolved/InvalidDeadend — map phase → a label + an on-brand tone; use ember ONLY for a punchy status like "Challenged"), the deadline (relative), proposer/fact counts, and the resolved option if Resolved. A `SectionHeader` intro. Click → detail.
- **Oracle detail** (`app/src/pages/OracleDetail.tsx` at `/oracles/:pubkey`): consume `fetchOracleDetail`. Show the oracle's full state (phase, deadline, options, bond pool, thresholds — laid out editorially, NOT a raw JSON dump), then sections for Facts (each: content-hash short, agreed/rejected, votes), Proposers (each: authority short, option, bond, correct/flipped/disqualified), AI Claims (each: submitter, option, model/params/io hashes short), and the Market if present (settle state). Use `Card`/`TriggerPreviewCard`/`AvatarBubble` (initials from pubkeys). Truncate pubkeys/hashes with a copy-on-click. Loading/error/not-found states.
- Delphi-faithful throughout (parchment canvas, chestnut-only fill, flat + hairlines, serif ≥20px, radii vocabulary, ember sparingly for 1–2 status punctuation moments, AA contrast). Responsive (375/1280). Accessible (landmarks, heading hierarchy, focus-visible, aria on status chips). Wire the routes; the NavBar links to `/oracles`.
- **Render verification:** since there's no standing deployment, render the UI with (a) MOCK data (a fixture set of decoded oracles) in a dev/story mode so the list/detail are visually reviewable offline, AND (b) if feasible, point a preview at the seeded surfpool from FA2 and confirm it lists the real seeded oracles. Verify with a headless browser at 375 + 1280 (screenshots + no console errors), like the slice-1 landing review.
- `pnpm --filter app typecheck` + `build` (+ `verify-css OK`) + `lint` green. Update `app/README.md` (the browse routes, the RPC/cluster config, how to point at a seeded surfpool). Commit `feat(app): oracle browser + detail views (read-only, Delphi style)`.

## Out of scope / deferred (next slice)
- WRITE flows: create oracle, propose, submit/vote fact, submit AI claim, finalize, open/settle challenge, claim/close/sweep — all signed by the wallet. (Builders exist in the SDK; wallet is wired here; the write UI + tx-confirm UX is the next slice.)
- A standing devnet/mainnet deployment (the app is configurable to point at one when it exists).
- Any program/SDK change (the app only consumes the built SDK).
- Real-time subscriptions / websocket account-change streaming (poll/refetch is fine for slice 2).
