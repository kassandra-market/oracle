**Goal:** Turn the thin JSON-paste challenge surface into a real challenge-market UI: **live market visualization** (pass/fail prices, TWAP→margin progress, twapEnd countdown, challenger position), **trade/crank/settle an OPEN market** (swap the conditional AMMs + crank the TWAP + settle-when-ready), and **client-side market composition** (open a challenge from the browser without pasting runner JSON — question → 2 conditional vaults → split → 2× createAmm → addLiquidity → openChallenge). ## Feasibility (from recon — all achievable without SDK change)
- **Amm decoder:** the SDK has NO v0.4 `Amm` decoder (only builders + PDAs). The app adds `decodeAmmV04` using the authoritative offsets in `programs/kassandra/src/cpi/metadao.rs:145-178` (disc `8ff5c8114ad6c487`; base_amount@115, quote_amount@123, oracle.last_updated_slot@131, oracle.aggregator u128@171, created_at_slot@9, start_delay_slots@219, AMM_MIN_LEN 227) — mirror the test-local `decodeAmmTwap` (`sdk/test/surfpool/challenge-market-e2e.test.ts:863-879`). Price = quote/base; TWAP = `aggregator/(last_updated_slot − (created_at + start_delay))` scaled by PRICE_SCALE 1e12 (the settle math, `metadao.rs:153-156`).
- **Market** already surfaced by `app/src/data/oracles.ts` `fetchOracleDetail` (passAmm/failAmm/kassVault/usdcVault/twapEnd/challengerUsdc/settled + escrow/holders).
- **Trade/crank/settle an open market:** `ammV04.swap` (`sdk/src/amm-v04/instructions.ts:201`), `ammV04.crankThatTwap` (`:246`), existing `buildSettleChallengeIxs` (`app/src/data/actions/challenge.ts:202`) — all accounts derive from the decoded Market + amm-v04 PDAs + user ATAs. crank rate-limited 1/150 slots; settle gated on `now/slot ≥ twapEnd`.
- **Compose:** the primitives all EXIST — `futarchy.initializeQuestion` (`sdk/src/futarchy/instructions.ts:129`), `futarchy.initializeConditionalVault` (`:158`, creates the vault + conditional mints), `futarchy.splitTokens` (`:222`), `ammV04.createAmm`/`addLiquidity`. Disc-verified against the challenge-market-e2e's real CPIs (`composeQuestion`/`composeVault` `:426-504`). The heavy part = the multi-tx orchestration + liquidity-seeding + observation/decimals math (mirror `composeMarket`/`buildPool` `:639-829` for the REAL bits; the E2E's setAccount pieces [oracle-owned holders, initial balances] become ordinary ATA-creates + split in production).

## CU1 delta (implemented)
- **`app/src/data/ammV04.ts`** (pure, no React): `decodeAmmV04(data)` — guards the 8-byte disc `8ff5c8114ad6c487` + `>= AMM_MIN_LEN` (227), reads base/quote mint (Pubkey @49/@81), base/quote decimals (u8 @113/@114), base/quote amount (u64 @115/@123), created_at/last_updated/start_delay slots (u64 @9/@131/@219), aggregator (u128 @171) — offsets exported as consts, mirroring `cpi/metadao.rs:145-178` + the test-local `decodeAmmTwap`. ## CU3 delta (implemented) — CLIENT-SIDE market composition (compose→open a challenge)
- **`app/src/data/actions/challengeCompose.ts`** (pure, no React): `buildComposeAndOpenChallengeIxs(args)` → `{ steps, composed }`. It COMPOSES the full MetaDAO v0.4 market client-side (the production equivalents of the SDK challenge-market-e2e's `surfnet_setAccount` cheatcodes become REAL ixs) and returns an ORDERED list of single-tx `ComposeStep`s: `question` (`futarchy.initializeQuestion`, binary, resolver == oracle PDA) → `kass-vault` / `usdc-vault` (`futarchy.initializeConditionalVault`, each creating the vault + its pass/fail conditional-token mints) → `fund-split` (idempotent ATA-creates for the **oracle-PDA-owned** pass/fail conditional-KASS holders [`ATA(oracle, condKassMint)` — replaces the E2E's `fabricateTokenAccountMint(passKass, oracle, 0)`] + the challenger's 4 conditional-token ATAs, then 2× `futarchy.splitTokens` splitting the challenger's KASS→pass+fail-KASS and USDC→pass+fail-USDC to seed both pools — replaces the E2E's `setTokenAccountAt(userBase, reserve×4)`) → `pass-pool` / `fail-pool` (`ammV04.createAmm` + an idempotent LP-ATA create + `ammV04.addLiquidity` — the LP-ATA create replaces the E2E's `setTokenAccountAt(userLp, 0)`; `add_liquidity` mints LP but does NOT create the ATA → `AccountNotInitialized`/0xbc4 without it) → `open` (RF4's `buildOpenChallengeIxs`, fed the composed accounts + `cvEventAuthority = futarchy.pda.vaultEventAuthority()` + `challengerUsdcSrc = ATA(challenger, usdcMint)`). The seed/TWAP math mirrors `buildPool` EXACTLY: `twapInitialObservation(base, quote) = quote·PRICE_SCALE(1e12)/base`, `MAX_OBSERVATION_CHANGE = (2^64−1)·1e12`, `twap_start_delay_slots = 0`, `add_liquidity(quote_amount = quoteReserve, max_base_amount = baseReserve)`. Because `split_tokens` mints EQUAL pass+fail from one underlying, splitting `baseReserve` KASS + `quoteReserve` USDC seeds BOTH pools. Defaults `DEFAULT_BASE_RESERVE = 100e9` (100 conditional-KASS, 9dp) / `DEFAULT_QUOTE_RESERVE = 100e6` (100 conditional-USDC, 6dp) / `DEFAULT_QUESTION_ID = fill(0x07)` (deterministic → same challenger⇒oracle ⇒ same market). Typed `ValidationError` on bad nonce / address / non-positive reserve / wrong-length question id. Each step carries a `computeUnits` hint (400k for question/vaults, 600k for fund+split, 1.4M for the pools/open).

On success → a chestnut "the market is live" line (CU1 panel + CU2 controls light up on refetch). Phase-gated by the caller (Challenge, no market) + `ConnectGate`'d. Delphi: chestnut submit/retry; ember reserved for the error lines only.
