An **oracle** in Kassandra is a single on-chain question with everything needed to resolve
it fixed at creation. It is represented by one `Oracle` account (`state.rs`, LEN = 360) and
created by the `create_oracle` instruction (`Ix::CreateOracle = 10`). Once created, its
configuration is immutable — the whole trust model rests on that.

## What a creator fixes at creation

`create_oracle` stands up a new oracle in `Phase::Proposal` with a future deadline, a
program-controlled KASS stake vault, a burned creation fee, and an immutable configuration.
The creator supplies (payload is exactly 25 bytes; `create_oracle.rs`):

<ParamField path="options_count" type="u8">
  The number of allowed **categorical options**. Must be at least 2, or creation fails with
  `InvalidOptionsCount`. Options are referenced by index (`0..options_count`).
</ParamField>

<ParamField path="deadline" type="i64">
  A unix timestamp that gates proposals: proposals are rejected before it
  (`DeadlineNotReached`), and it must be in the future at creation (`InvalidDeadline`). The
  proposal window opens at the deadline and runs for `proposal_window` seconds.
</ParamField>

<ParamField path="twap_window" type="i64">
  The per-oracle TWAP duration (seconds) used by challenge markets: a market's settle is
  allowed only after `market.twap_end = open_time + oracle.twap_window`. Must be positive.
</ParamField>

<ParamField path="kass_mint / usdc_mint" type="Pubkey">
  The KASS mint (must match the Protocol's canonical KASS) used for all bonds/stakes, and
  the USDC mint used as challenge-market collateral.
</ParamField>

The `create_oracle` payload no longer carries the question itself. The **question, option
labels, and interpretation** live on-chain in a companion `oracle_meta` PDA
(`[b"oracle_meta", oracle]`), written by the `write_oracle_meta` instruction
(`Ix::WriteOracleMeta = 23`) appended to the **same transaction** so the metadata lands
atomically with the oracle. `oracle_meta` stores, length-prefixed, the plaintext **subject**
(the question), the option **labels**, and a **`uri` plus 32-byte `uri_hash`** that binds an
extended off-chain JSON (`{version, subject, options, promptTemplate, interpretation?,
category?, createdAt}`). That JSON's `promptTemplate`/`interpretation` is the immutable rule
for how evidence maps to an answer — the runner fetches the `uri` and refuses to resolve
unless `sha256(json) == uri_hash`.

Everything else — the bonding and slashing behavior, phase windows, fact-quorum threshold,
challenge-market thresholds and fees, and reward weights — is **snapshotted** from the
global `Protocol` account onto the `Oracle` at creation.

<Note>
  The snapshot is a deliberate anti-governance-attack measure. Downstream processors read
  these params **from the Oracle**, not from the live `Protocol`, so a mid-dispute
  governance change to `Protocol` cannot move the goalposts on an oracle already in flight
  (`create_oracle.rs`; the frozen fields include `threshold_*`, `market_threshold_*`,
  `flip_slash_*`, `phase_window`, `proposal_window`, reward weights, and challenge fees).
</Note>

## The burned KASS creation fee

Creating an oracle costs a **dynamic KASS fee that is burned**, not paid to anyone. The fee
tracks an exponential moving average of recent creation activity (`fee.rs`):

```
fee = fee_for_ema(decay_fee_ema(fee_ema, last_creation_unix, now))
```

At genesis the EMA is zero, so **creation is free** — you can bootstrap into the system and
earn KASS via emissions without having to acquire it first. As creations pile up the EMA
rises (each creation bumps it by `FEE_EMA_INCREMENT`), and it decays with a one-day
half-life (`FEE_EMA_HALFLIFE_SECS = 86_400`) as activity relaxes. The burn is the
deflationary counterweight to work-to-earn emissions; there is **no USDC floor**. See
[Economics](/concepts/economics) for the exact EMA math.

## Binary and categorical questions

Kassandra answers **binary and categorical** questions only — a finite set of discrete
options. It does not do continuous/numeric resolution. The categorical analogue of a median
is the **mode / plurality**: after a dispute, each *surviving* proposer casts exactly one
vote for their value, and the option with the strictly-highest count wins (`plurality.rs`,
returning `Winner(u8)`, `Tie`, or `NoSurvivors`).

<Info>
  Plurality is **one proposer, one vote** — not stake-weighted. Option index 0 is a valid
  winning option and is distinct from "no votes." A tie between two or more options yields
  no winner and sends the oracle to an [Invalid dead-end](/concepts/dead-end-settlement).
</Info>

## The `Oracle` account at a glance

The account carries the fixed config, the current `phase`, live counters, and the
resolution stamps written at the end. A few fields worth knowing early:

| Field | Meaning |
|---|---|
| `phase` | current `Phase` as a `u8` — the oracle's position in the [lifecycle](/concepts/lifecycle) |
| `stake_vault` | PDA token account (`[b"vault", oracle]`) holding all KASS bonds and stakes |
| `deadline` / `phase_ends_at` | proposal deadline and the end of the current window |
| `options_count` / `resolved_option` | number of options; the final winner (valid only when `Resolved`; `0xFF` on `InvalidDeadend`) |
| `proposer_count` / `surviving_count` | total proposers, and those not yet disqualified |
| `dispute_bond_total` | sum of proposer bonds, fixed at dispute start — the fact-quorum denominator |
| `bond_pool` | accumulated slashed KASS that funds rewards on resolution |

The full field-by-field layout, PDA seeds, and byte offsets are in the protocol reference.
