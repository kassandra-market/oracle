The **kassandra-market** program is a thin Pinocchio (no-Anchor) wrapper that
composes and verifies MetaDAO v0.4 `conditional_vault` + `amm` accounts,
program-signs the value-moving CPIs, and defers resolution to the Kassandra
oracle. It never custodies value on the trading hot path.

- **Program ID:** `FEGNHWAB7kc7VC9CCwbvVPsv4Jykz2r2WQ758V4xCT9S`
  (`programs/markets/src/lib.rs:17`).
- **Entrypoint:** `process_instruction` â†’ `processor::process`, which reads the
  first byte of `instruction_data` as the [instruction discriminant](/market-protocol/instructions)
  and dispatches to one processor per file (`processor/mod.rs`).
- **External programs bound:** conditional_vault
  `VLTX1ishMBbcX3rdBWGssxawAo1Q2X2qxYFYqiGodVg`, amm v0.4
  `AMMyu265tkBpRW21iGQxKGLaves3gKm2JcMUqfXNSpqD`.

## Account-type tags

Every account is a `#[repr(C)]`, `Pod + Zeroable` struct, fully packed with
explicit padding, whose **first byte is an `AccountType` tag** to reject
type-confusion (`state.rs:17`). A freshly `CreateAccount`'d account carries
`Uninitialized` (0) until it is stamped.

| Variant | Value | Account |
| --- | --- | --- |
| `Uninitialized` | 0 | (zeroed, unstamped) |
| `Config` | 1 | the governed singleton |
| `Market` | 2 | a per-outcome binary sub-market |
| `Contribution` | 3 | a per-contributor stake |

## The security model

The program is audited-in-spirit against the
[Solana Security Standard](https://github.com/Copenhagen0x/solana-security-standard).
Because there is no Anchor, every check is hand-rolled in `processor/guards.rs`:

### init_config is gated to the upgrade authority

The `[b"config"]` address is deterministic and known at deploy time, so an attacker
could try to front-run genesis and seize `Config.authority` (the futarchy
governance key). `init_config` defends against this two ways
(`processor/init_config.rs`):

- **Bootstrap authorization:** the caller must be the program's **on-chain upgrade
  authority**, proven by reading the BPF-Upgradeable-Loader `ProgramData` account;
  otherwise `NotUpgradeAuthority`.
- **Create-or-adopt:** rather than a plain `CreateAccount` (which fails on any
  pre-funded account and would brick genesis), it tops the balance up to
  rent-exempt, then `Allocate`s + `Assign`s the PDA under the program seeds â€” so a
  pre-funded singleton is adopted, not blocked. Re-init is rejected by the
  account-type **tag**, not lamports.

The same create-or-adopt idiom protects the three Market-PDA-owned token accounts
in `activate`, so a lamport donation to a derivable PDA can never brick activation.
